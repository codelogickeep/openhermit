
### 📋 1、产品需求文档 (PRD) - Phase 1 (钉钉版)

**1. 产品定位**
将本地机器上运行的强交互 CLI 应用（以 Claude Code 为主），通过 PTY 劫持技术，无缝接入移动端办公通讯软件（钉钉），实现安全的远程跨端 AI 编程与指令下发。

**2. 核心模块与功能需求**

* **模块一：通道接入层 (Channel Gateway)**
* **需求 1.1：** 支持通过钉钉企业内部机器人的 AppKey 和 AppSecret 建立 Stream 长连接。
* **需求 1.2：** 支持接收钉钉单聊文本消息，并将其作为 `stdin` 注入本地伪终端。
* **需求 1.3：** 支持向钉钉推送纯文本消息（防抖合并后）。
* **需求 1.4：** 支持向钉钉推送 ActionCard（用于权限确认），并处理按钮回调事件。


* **模块二：核心桥接层 (PTY Bridge)**
* **需求 2.1：** 能够通过 `node-pty` 派生子进程，并继承宿主机的所有系统环境变量（极其重要，确保 MCP 工具链生效）。
* **需求 2.2：** 内置命令支持：识别特定的斜杠命令，如 `/cd [目录路径]` 用于切换当前 Agent 的工作目录，`/restart` 用于重启崩溃的子进程。


* **模块三：数据清洗与调度 (Stream Purifier)**
* **需求 3.1：** 实时拦截 `stdout` 流，使用正则或依赖库剥离 ANSI 控制字符。
* **需求 3.2：** **高优先级：** 具备“拦截器”机制，精准匹配类似 `Do you want to run this command? (y/n)` 或 `Allow (y/N)` 的特征字符串，触发 HITL（人在回路）卡片推送。
* **需求 3.3：** 文本防抖（Debounce），例如设置 1000ms 的窗口期，将短时间内的零碎输出拼接成一段完整的长文本再推送到公网，避免触发 IM 接口限流。



**3. 第一阶段非目标 (Out of Scope)**

* 暂不支持微信/企业微信（避免引入复杂的内网穿透配置）。
* 暂不支持复杂的多会话并发隔离（Phase 1 仅支持单例模式，即一部手机控制一个本地主进程）。

### 🏗️ 2. 概要技术架构文档 (Architecture)

**1. 技术栈选型**

* **运行环境:** Node.js (v18+)
* **核心引擎:** `node-pty` (伪终端派生)
* **通道 SDK:** `dingtalk-stream-sdk-nodejs`
* **辅助工具:** `strip-ansi` (清理终端乱码), `lodash.debounce` (流式防抖)

**2. 核心架构设计图**

```text
[钉钉手机端 / PC端]
       |
  (JSON / WebSocket 长连接)
       |
+------|-------------------------------------------------------+
|   [ 钉钉 Stream Client ] (维持心跳，接收指令与回调)              |
|      |                                                       |
|   [ Command Parser ] (分离内置命令与透传文本)                    |
|      |                                                       |
|   [ Stream Purifier & HITL Detector ] (核心逻辑大脑)         |
|   - Regex 匹配拦截 "(y/n)" 触发 ActionCard                       |
|   - Strip-ansi 剥离 Loading 动画和颜色符                           |
|   - Debounce Buffer 防抖拼装长文本                                 |
|      |                                                       |
|   [ PTY Engine ] (node-pty)                                  |
+------|-------------------------------------------------------+
       | (纯净的 stdin / 脏的 stdout 包含控制符)
       v
[ claude 进程 ] (Claude Code CLI)
       |
[ 继承的 process.env ] --> [ 本地各类 MCP Servers ]

```

**3. 核心难点与攻克方案**

* **难点 A：终端重绘导致的乱码。** Claude Code 在思考时，光标会原地转圈。这会在 `stdout` 产生大量的 `\b` (退格) 和回车符。
* **解决：** 在 `Stream Purifier` 层，不仅要用 `strip-ansi` 去颜色，还需要维护一个行缓冲（Line Buffer），过滤掉频繁覆盖同一行的特殊字符，只将稳定换行的内容送入防抖队列。


* **难点 B：等待用户输入时的进程阻塞。** 当本地跑 `npm i` 或 `git commit` 需要用户输入时，终端流会突然停止，没有换行符。
* **解决：** 不依赖换行符进行推送。设置超时阈值（例如 500ms 内没有新流输出），结合 `stdout` 尾部是否匹配 `(y/n)` 正则，来判断当前是否处于“等待授权”状态。
